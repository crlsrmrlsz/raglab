Function: should_use_map_reduce(query) calls classify_query()

The LLM responds with a single word: "local" or "global".
Local Path — Graph Traversal
L1 Entity Extraction
Embed query, find top-K entities via Weaviate HNSW
extract_query_entities(query, top_k=5)
Embeds the query with embed_texts(), then runs vector similarity against the Entity collection in Weaviate. Returns entity names like ["dopamine", "motivation", "reward"].
L2 Graph Traversal
2-hop Neo4j traversal from matched entities
find_entity_neighbors(driver, entity, max_hops=2)
Starting from matched entities, traverses the knowledge graph in Neo4j to discover related entities and their source_chunk_ids. Captures structural context that embedding similarity alone would miss.
L3 Chunk Retrieval
Batch-fetch source chunks from Weaviate by ID
fetch_chunks_by_ids(chunk_ids, collection)
Uses Weaviate ContainsAny filter — not vector search. Retrieves full chunk text for all graph-discovered chunk IDs in a single batch call.
L4 Rank by Combined Degree
Hub entities = more informative chunks (Microsoft design)
combined_degree = start_degree + neighbor_degree
Chunks reached via well-connected "hub" entities score higher. Sorted descending — top-k returned. This is Microsoft GraphRAG's relationship prioritization strategy.
L5 Community Context
Enrich with community summaries by entity membership
retrieve_community_context_by_membership(entities, driver)
Queries Neo4j for community IDs containing the matched entities, then fetches those community summaries from Weaviate. Adds thematic background to the answer context.
Global Path — DRIFT Search
G1 Embed Query
Convert query to embedding vector
embed_texts([query])[0]
Same embedding model used during indexing (text-embedding-3-large). The vector will be compared against pre-computed community summary embeddings.
G2 HNSW Top-K Communities
Retrieve 20 most relevant L0 communities via vector search
query_communities_by_vector(embedding, top_k=20)
Weaviate's HNSW index finds the most relevant communities in O(log n) — replacing the original map-reduce which processed ALL communities at O(n). Only L0 (coarsest) communities are searched.
G3 Primer Folds (Parallel)
4 parallel LLM calls, each processing 5 communities
_primer_fold_async(query, fold, model)
Top-20 communities split into 4 folds of 5. Each fold runs concurrently via asyncio.gather(). Each LLM call produces an intermediate answer with a relevance score [0-10].

G4 Filter + Score
Remove irrelevant answers, sort by score descending
Filters out "Not relevant" responses and answers with score=0. Remaining intermediate answers are sorted by relevance score. Typically 2-4 folds contribute meaningful content.
G5 Reduce (Final Synthesis)
Single LLM call synthesizes all intermediate answers
Uses DRIFT_REDUCE_PROMPT to merge intermediate answers into one coherent response. Removes redundancy and preserves data citations. Total: ~5 LLM calls (4 primer + 1 reduce) vs ~1000 for old map-reduce.
Answer Final Response
Local: LLM generates answer from ranked chunks + community context
Global: DRIFT result returned directly (already synthesized)
Local path: generate_answer(query, chunks, graph_context) — formats ranked chunks as numbered context, adds community summaries as background, calls LLM to produce final answer with source citations [1], [2], etc.

Global path: The DRIFT reduce step already produces the final answer. It's packaged as a DriftResult with metadata (communities used, timing, LLM call count) and returned directly — no additional LLM call needed.
